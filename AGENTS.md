## 3. Canonical Workflow

Every session must follow this order:

```bash
bash init.sh                          # 1. bootstrap environment
python enrich.py                      # 2. regenerate enriched dataset
python verify.py                      # 3. confirm 34/34 checks pass
```

**Never proceed to modelling if verify.py exits with code 1.**

If a parameter change is made to `enrich.py`, always re-run both scripts
and confirm the full check suite still passes before committing anything.

---

## 4. What Agents Are Allowed To Do

- Modify synthesis parameters in `enrich.py` (constants block only) with
  documented justification added to `memory.md`
- Add new checks to `verify.py` — never remove or weaken existing ones
- Update `README.md` to reflect parameter changes
- Append to `memory.md` and update `context.md`
- Build downstream modelling scripts (OLS regression, time-series) using
  `academic_performance_enriched.csv` as input

---

## 5. What Agents Must NOT Do

- Modify `academic_performance_dataset_V2.csv` — it is the immutable source
- Change `--seed` default from 42 without explicit human instruction and a
  `memory.md` entry explaining why
- Remove the `BLEND_ALPHA` mechanism or revert to pure `f(CGPA)` synthesis
  — this was a deliberate scientific fix (see memory.md §2)
- Redefine `Previous_GPA` back to `CGPA100` — this was a deliberate fix (see memory.md §3)
- Redefine `Course_Load` back to credit units — it must remain course count (see memory.md §4)
- Claim that OLS results from this dataset constitute empirical evidence —
  they are illustrative only (see Limitations L1–L3 in README.md)
- Silently weaken or drop any of the 34 verify.py checks

---

## 6. Parameter Change Protocol

When changing any constant in `enrich.py`:

1. State the scientific reason for the change
2. Re-run `enrich.py` and `verify.py` — confirm 34/34 pass
3. Add an entry to `memory.md` under the relevant section
4. Update the corresponding section in `README.md`
5. Update `context.md` with what changed and why

---

## 7. Scientific Red Lines

These constraints are non-negotiable and must survive across all sessions:

| Constraint | Reason |
|---|---|
| BLEND_ALPHA = 0.60 (or justified alternative) | Endogeneity mitigation |
| Previous_GPA = mean(CGPA100, CGPA200, CGPA300) | Arithmetic overlap reduction |
| Course_Load = number of courses (8–20) | Correct variable definition |
| verify.py must exit 0 before any downstream work | Data integrity gate |
| Limitations L1–L6 must appear in any paper using this data | Scientific honesty |

---

## 8. Disambiguation Rules

- "GPA" in this project always means the 5.0-scale CGPA used at Mountain Top University
- "Course Load" always means **number of courses per semester**, not credit units
- "Previous GPA" always means the lagged three-year cumulative average, not CGPA100
- The dependent variable is `CGPA` (overall cumulative) unless explicitly reframed to `CGPA400`
- Programme codes in the dataset (CIS, MAT, etc.) are the institution's actual codes,
  not the CS/Maths department labels (CSC, SWE, CYB, MATH) used in the paper's framing

---

## 9. Open Questions for Human Decision

- Whether to run separate models per programme tier or a single pooled model
